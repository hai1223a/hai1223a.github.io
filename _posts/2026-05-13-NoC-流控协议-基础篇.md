## 通信的载体：Message、Packet、Flit 和 Phit

NoC 中通信的信息共有 4 个抽象层次：

* **消息（message）**：从用户视角看到的一次通信的全部信息，没有大小限制。
* **包（packet）**：message 被分为若干个固定大小的 packet，是路由和排序的基本单元，包含 header（路由和控制信息）、body（数据）和 tail（错误代码或数据），通常为 64B-64KB。
* **微片（flit）**：packet 拆分为 flit，是流控的基本单元，包含 type（类型）、VCID（虚通道）和数据，通常为 2B-64B。
* **物理微片（phit）**：flit 需要多个物理时钟周期传输，每个时钟周期传输的信息就是 phit，通常为 1b-64b。

## 基于 Message 粒度的流控

**Circuit Switching（电路交换）**以 message 为粒度预先分配网络资源，它不需要 buffer，所以一旦 setup 之后就是无阻塞的。其一共有四个阶段：

1. source 先发送一个很小的 **probe / setup flit**，沿着 route 前进，并预留经过的 links。
2. 当 probe 到达 destination 后，destination 返回一个 **acknowledgement flit**，通知 source：路径已经建立完成。
3. source 收到 acknowledgement 后，开始发送真正的 **data message**。由于路径已经提前预留，data 可以快速通过各个 router，不需要每一跳重新竞争资源。
4. data 传输完成后，source 发送 **tail flit**，用于释放之前预留的 links。

但是这种流控十分浪费带宽资源，并会引起严重的阻塞。如图所示，一个使用 X-Y DOR 路由的 mesh 网络，在 t=4 时完成了 setup，而 t=5 时 S2 想要通信却被阻塞，一直等到 t=19 时 S0 释放了链路资源才开始 setup。图中时空图灰色部分全部都是闲置状态。

## 基于 Packet 粒度的流控

### Store-and-Forward（存储转发）

**Store-and-Forward** 以 packet 为粒度分配资源，它需要维护 buffer，所以只有当下游节点的 buffer 有足够容纳一个 packet 的空间时，才会开始传输。它的核心规则是：

**每个 router 必须先完整接收整个 packet，然后才能把这个 packet 转发到下一跳。**

**优点：实现简单。缺点 1：每跳延迟大。缺点 2：buffer 需求大。**

如图，t=5 时 core1 接收到 core0 传来的 head flit，但是必须等到 t=10 时才能够发送给 core2。


![image-20260512162701313](/images/26.jpg)
<p class="image-caption">图 3：Store-and-Forward 存储转发流控示例</p>

### Virtual Cut-through（虚跨步交换）

**Virtual Cut-through** 改进了 Store-and-Forward 的延迟问题。它的核心规则是：

**当前 router 不必等完整 packet 收完，只要下一跳有足够 buffer 容纳整个 packet，head flit 就可以先继续前进。**

**优点：延迟明显降低。缺点1：仍然按 packet 粒度分配 buffer。缺点 2：大 packet 时 buffer 成本仍然高。**

## 基于 Flit 粒度的流控

### Wormhole Flow Control（虫洞）

**Wormhole** 把 packet 拆成多个 **flit**，并以 flit 为单位分配 buffer 和 link bandwidth。核心规则是：

**只要下一跳有足够 buffer 容纳当前 flit，这个 flit 就可以继续前进，不需要等整个 packet 到齐。**


它相比 **store-and-forward** 和 **virtual cut-through** 的主要优势是：**buffer 需求小**。因为 router 不需要准备能容纳整个 packet 的大 buffer，只需要若干个 flit buffer 即可。

但它的问题是：**一个 packet 可能横跨多个 router**。如果 head flit 在前方被阻塞，后面的 body/tail flits 也会停住，导致该 packet 已经占有的部分 link / channel 处于 **idle but unavailable** 状态。这也是经典的 head-of-line blocking 问题。

### Virtual Channel Flow Control（虚通道）

**Virtual Channel, VC** 是在同一个 physical channel 上划分多个逻辑队列。它不是增加物理链路，而是在一个 input port 上增加多个独立 queue。虚通道解决的是 **head-of-line blocking** 问题，它可以和之前提到的所有流控技术结合。比如：

* **虚通道电路交换**，一个 message 预留的是一系列 VCs，而不再是物理 link。多个 VCs 以 cycle-by-cycle 的方式仲裁映射到物理 link。
* **虚通道存储转发**，就是设置多个 packet buffer queues，每个 queue 对应一个 VC。VCs 以 packet-by-packet 的方式仲裁映射。
* **虚通道虚跨步交换**，与虚通道存储转发的区别是，VCs 以 flit-by-flit 的方式仲裁映射。

但是现在常说的虚通道流控，通常还是虚通道与 wormhole 相结合。其核心规则是：**VC 是 packet 级占用；physical link 是多个 VC cycle-by-cycle 仲裁共享。**

> 这里提到的 cycle-by-cycle 是指每个周期进行一次虚通道仲裁，而 packet-by-packet 是指每次通信完一个 packet 进行一次虚通道仲裁，这表示了仲裁的粒度。

有如下案例：Packet A 和 Packet B 一开始都在 Node 0，且都想使用同一条 east output physical channel；但它们位于不同的 VC 中，A 占用 VC0，B 占用 VC1。A 的 head flit 先进入 Router 1 并占用 VC0，随后 B 的 head flit 也可以通过同一条 physical link 分时进入 Router 1，并占用 VC1。之后 A 想继续前往 Router 4，但由于下一跳没有空闲 VC，A 被阻塞在 VC0 中；如果没有 virtual channel，B 会因为排在 A 后面而一起被阻塞。但在 VC flow control 中，B 位于独立的 VC1，只要它去往 Router 2 的资源可用，就可以继续前进。因此，virtual channel 的作用是把一个 input port 拆成多个逻辑队列，使被阻塞的 packet 只占住其中一个 VC，而不会阻塞其他 VC 中可以继续前进的 packet；同时多个 VC 仍然共享同一条 physical link，通过 switch allocation 按 cycle 分时使用。

## 无死锁流控

前文提到可以使用无死锁路由算法，但是其实也可以设计无死锁的流控策略，这样就可以使用任意路由算法了。

### Dateline and VC Partitioning（日期变更线和VC分区）

这两个技术是利用 VCs 来实现无死锁的。**Dateline** 的思想是将依赖环人为划定一条线，如图所示，在 dateline 之前只能使用 VC0，在 dateline 之后只能使用 VC1，这样就打破了依赖环。

<img src="/images/29.jpg" alt="Dateline and VC partitioning" style="zoom: 50%;" />
<p class="image-caption">图 6：Dateline 与 VC Partitioning 打破依赖环的示例</p>

更加普遍的 **Partitioning** 也是类似思想。它将 VCs 分为不同的 class，并规定对应的 packet 只能在 class 中的 VCs 内进行选择。比如使用随机选择 X-Y DOR 和 Y-X DOR 的遗忘路由，可以将 X-Y DOR 放到 class0，将 Y-X DOR 放到 class1，这样就不会形成死锁了。

这两个方法的优点是**实现简单**，但是**VC 利用率可能低**。

### Escape VCs（逃逸通道）

该方法核心在于保留一条 VC0 使用无死锁路由，而其他 VCs 使用任意路由。这样当其他 VCs 形成依赖环时，只要仲裁是公平的，某个 packet 迟早能进入 VC0。一旦进入 VC0，它就走一条保证无死锁的路径。这个 packet 前进后，会释放自己原先占用的 VC / buffer，其他 packet 也就有机会继续前进。

### Bubble Flow Control（气泡流控）

通常针对 ring、torus 这样自带环的拓扑结构。这个技术不依赖 VC，通常与虚跨步交换配合使用。其核心在于，在注入或者转换维度时，需要检查注入节点的 buffer 是否存在两个 packet 的空闲空间。如图所示，只有 P1 能够注入这个环，而 P0 和 P2 不行。

> 注入就是从本地节点有 packet 进入本地 router
>
> 维度转换就是本来位于 X 维度环的 packet，想要向 Y 维度移动了（反之亦然）

## 缓冲区背压

NoC 不允许丢包，所以需要背压技术来暂停 flit 的发送。缓冲区背压的控制粒度取决于流控技术：存储转发和虚跨步交换以 packet 为粒度，虫洞交换和虚通道以 flit 为粒度，电路交换没有缓冲区。

### 基于 Credit 的缓冲区背压

上游维护一个 **credit count**，表示下游还有多少可用 buffer slot。

基本过程：

```
下游释放一个 buffer
→ 返回一个 credit 给上游
→ 上游 credit count +1

上游发送一个 flit / packet
→ 消耗一个 credit
→ 上游 credit count -1
```

只有当 credit count 大于 0 时，上游才能继续发送。

### 基于 On/Off 的缓冲区背压

下游不告诉上游具体还有多少 buffer，只告诉上游：

```
on  → 可以继续发
off → 暂停发送
```

当下游 free buffer 数量低于某个阈值时，发送 **off signal**，阻止上游继续发送；当 free buffer 数量恢复到另一个阈值以上时，发送 **on signal**，允许上游继续发送。

## 实现

流控协议实现的开销主要是路由器微架构的开销，以及 router 间通信的布线开销。

**Buffer turnaround time（缓冲周转时间）** 指的是：同一个 **buffer（缓冲槽）** 被连续两个 **flits（流片）** 复用之间的最小空闲时间。而使用两种缓冲区背压机制得到的缓冲周转时间不一样，如果 buffer 的数量不能覆盖掉这段时间，那么即使没有任何竞争，整个流水线也会因此阻塞。

### 基于 Credit 的缓冲区背压

以如下通信过程为例：

```
Node 0  ----data flit---->  Node 1
Node 0  <----credit-------  Node 1
```

$$
\begin{aligned}
\text{缓冲周转时间} =& \ \text{data flit propagation delay（数据 flit 传播延迟）} + \\
&\text{credit delay back（credit 返回延迟）} +\\
&\text{pipeline delay（流水线延迟）}
\end{aligned}
$$

也就是包含了下图的后四段延迟时间，假设

> data flit propagation delay（数据 flit 传播延迟）= 1 cycle
>
> reverse signaling propagation delay（反向信号传播延迟）= 1 cycle
>
> buffer backpressure signal pipeline delay（反压信号流水延迟）= 1 cycle
>
> router pipeline（路由器流水线）= 3 cycles

那么这个缓冲周转时间就是 6 cycle，需要下游节点有 6 个 buffer 才能完全抵消。

![image-20260513135338604](/images/31.jpg)
<p class="image-caption">图 8：Credit-based 背压机制中的缓冲周转时间</p>

但是这种机制需要更多的信号线，总共需要 $n\log B$，其中 n 表示虚通道数量，B 表示一个虚通道 buffer 数量。

### 基于 On/Off 的缓冲区背压


$$
\begin{aligned}
\text{缓冲周转时间} =& \ 2 \times \text{on/off signal propagation delay（on/off 信号传播延迟）} + \\
& \text{data flit propagation delay（数据 flit 传播延迟）} + \\
& \text{pipeline delay（流水线延迟）}
\end{aligned}
$$

下方图中两条时间线是同时发生的，这里 2 倍 on/off 传输延迟我觉得应该是留余量。同样按照上面的实验数据，那么这里还是主要覆盖后面四段时间，具体延迟是 8 cycle。

<img src="/images/32.jpg" alt="On-off backpressure" style="zoom:67%;" />
<p class="image-caption">图 9：On/Off 背压机制中的缓冲周转时间</p>

但是信号线更少，只需要 $n$ 根，n 为虚通道数量。
